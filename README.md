# Небольшая шпаргалка по GIT

### Последовательность основных команд Git для управления версиями проекта:

1. Инициализация репозитория:
   
   `git init`
   
   Эта команда создаёт новый репозиторий Git в текущей директории.
2. Добавление файлов в индекс:
   
   `git add <путь_к_файлу_или_каталогу>`
   
   Например, `git add .` добавляет все новые и изменённые файлы в индекс.
3. Совершение коммита:
   
   `git commit -m "Сообщение о коммите"`
   
   Напишите краткое и информативное сообщение о том, что было изменено.
4. Просмотр состояния репозитория:
   
   `git status`
   
   Показывает текущие изменения и состояние файлов в репозитории.
5. Просмотр истории коммитов:
   
   `git log`
   
   Отображает историю коммитов с деталями.
6. Создание ветки:
   
   `git branch <имя_ветки>`
   
   Создаёт новую ветку.
7. Переключение на другую ветку:
   
   `git checkout <имя_ветки>`
   
   Переключает на указанную ветку.
8. Слияние веток:
   
   `git merge <имя_ветки>`
   
   Сливает изменения из указанной ветки в текущую ветку.
9. Добавление удалённого репозитория:
   
   `git remote add origin <URL_репозитория>`
   
   Добавляет удалённый репозиторий с указанным URL.
10. Отправка изменений в удалённый репозиторий:
   
   `git push origin <имя_ветки>`
   
   Отправляет изменения из локальной ветки в удалённый репозиторий.
11. Получение изменений из удалённого репозитория:
   
   `git pull origin <имя_ветки>`
   
   Получает изменения из удалённого репозитория и объединяет их с локальными изменениями.



## Основы работы с Git

Git — это распределённая система контроля версий, которая позволяет отслеживать изменения в файлах и сотрудничать с другими разработчиками.

### Основные понятия

#### HEAD
HEAD в Git — это указатель на последнюю коммит-версию в ветке, с которой вы работаете. По сути, HEAD указывает на текущий рабочий коммит, и при каждой новой фиксации (коммите) HEAD перемещается на этот новый коммит.

#### Хеш
Хеш в Git — это уникальный идентификатор, который присваивается каждому коммиту. Этот идентификатор представляет собой строку символов, которая генерируется на основе содержимого коммита. Хеш позволяет однозначно идентифицировать каждый коммит и обеспечивает целостность данных.

#### Лог
Лог в Git — это история всех коммитов в репозитории. Вы можете просмотреть лог с помощью команды `git log`, которая покажет список всех коммитов с их хешами, авторами, датами и сообщениями.

#### Статус файла
Статус файла в Git указывает на его текущее состояние относительно репозитория. Файл может иметь следующие статусы:
- неотслеживаемый (untracked) — файл не отслеживается Git;
- изменённый (modified) — файл был изменён, но ещё не добавлен в индекс;
- индексированный (staged) — файл добавлен в индекс и готов к коммиту;
- зафиксирован (committed) — файл был зафиксирован в репозитории.

### Правила оформления коммита

При оформлении коммита важно следовать определённым правилам, чтобы обеспечить ясность и понимание истории изменений:
1. **Краткость и ясность**: сообщение коммита должно быть кратким и понятным. Оно должно описывать, что было изменено, без лишних деталей.
2. **Первая строка**: первая строка сообщения коммита должна быть короткой (не более 50 символов) и описывать суть изменений.
3. **Пустая строка**: после первой строки должен следовать пустой ряд, а затем более подробное описание изменений (если необходимо).
4. **Глаголы**: начинайте сообщение коммита с глагола в настоящем времени, например, «Добавить», «Исправить», «Удалить» и т. д.
5. **Единообразие**: старайтесь придерживаться единого стиля оформления сообщений коммитов, чтобы они были легко читаемыми и понятными для всех участников проекта.

Пример оформления коммита:
```
Добавить функцию расчёта среднего значения

Эта функция позволяет рассчитывать среднее значение для списка чисел. Она принимает список чисел в качестве аргумента и возвращает среднее значение.
```
